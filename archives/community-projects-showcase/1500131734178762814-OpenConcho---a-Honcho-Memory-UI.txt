    I've tried them all, honestly. Mem0, QMD, Mempalace, and now Honcho. I freakin' love Honcho, but I hate saying it. I digress.
    
    Mem0 (you're using a lower model to store facts a higher model already has synthesized)
    MemPalace (Not very mac friendly atm, but getting there and has promise)
    QMD (Fairly, still use it, nothing beats markdown, but Honcho synthesizes in the background)
    
    I figured out how to self-host honcho not only on docker compose, but kubernetes. I have it humming on my home lab and served via TailScale.

    I figured why should I have to pay for Honcho to have a UI?
    
    I'd love your feedback, support, and contributions if you have time 🙂
    
    https://github.com/offendingcommit/openconcho
    
    Please let me know if you have any questions. It's still a baby, so hoping that the usefulness aids you all.

[2026-05-03T03:53:34.660000+00:00] jwalkersf7273 (id=1500344545328500736)
    from my docs:
    honcho-rep-context Hermes plugin
    
    `pre_llm_call` hook that injects a Honcho per-rep peer card on the first turn
    of every session. Maps the Slack `user_id` (passed by the Hermes gateway as
    `sender_id`) to a Honcho `peer` in the configured workspace.
    
    ## What it does
    
    | Stage | Behavior |
    | --- | --- |
    | First turn | Look up Honcho peer = `sender_id` in workspace `deepswag` (or `HONCHO_WORKSPACE`). Auto-create the peer if missing. Fetch `peer.get_representation()`. Inject as a context block on the user message. |
    | Subsequent turns | No-op. Per-turn recall is hindsight-context's job. |
    | Empty card | Return `None`. Empty representation is honest — new peers won't have a representation until Honcho's deriver processes observations. Do NOT fabricate filler. |
    | Honcho down | Return `None`, log warning. NEVER block the turn. |
    | `sender_id` empty | Return `None`. Log at DEBUG so stale gateways surface. |
    
    This plugin is **complementary to**, not a replacement for, Hermes's bundled
    `memory.provider: honcho` integration. The bundled provider handles
    end-of-session writes (conclusions extraction, message persistence). This
    plugin handles the per-rep first-turn peer-card injection that the memory
    provider doesn't do natively.
