    I’m trying to design the memory/context layer for an "AI-native" company and would love advice from people who have built something similar.
    
    The goal is not just “chatbot memory.” I’m thinking about a real company-wide second brain / agentic coworker setup.
    
    **Use case:**
    We need to store and continuously update context across multiple users, projects, clients, and internal agents. 
    The knowledge sources would include things like:
    - Slack discussions and decisions
    - email threads
    - client call transcripts
    - product analytics alerts
    - GitHub commits / PRs
    - internal docs
    - client reports
    - research notes
    - maybe CRM/project-management data later
    
    The system should update itself after each new piece of knowledge, e.g. after a Slack thread, email, call, new commit, analytics alert, etc.
    
    I’m currently comparing Honcho and Supermemory.
    
    So my rough hypothesis is:
    - Supermemory = company knowledge brain/retrieval layer
    - Honcho = user/agent memory layer
    
    
    But I’m not sure if this is overengineering.
    
    Some questions:
    1. Has anyone used Honcho or Supermemory in production for a real multi-user/team/company use case?
    
    2. If the goal is a “company second brain” connected to Slack, email, calls, GitHub, docs, and analytics — which architecture would you choose?
    
    3. How do you handle “latest truth”? For example, Slack says one thing, a later client email says another, and a signed contract says a third thing. Do you encode source priority manually?
    
    4. How do you prevent the memory layer from becoming polluted with speculation, outdated context, or low-quality Slack noise?
    
    But maybe I’m making this too complex too early.... idk
    Would love to hear how others are thinking about memory architecture for "AI-native" companies
