    Mnemosyne Update: Critical Fixes for All Users
    
    Pushed two important fixes to Mnemosyne today that affect anyone using or upgrading to the BEAM architecture:
    
    1. Legacy Database Migration Path
    The BEAM upgrade changed the canonical database path, which meant users coming from pre-BEAM versions silently lost access to their existing memories. We shipped scripts/migrate_from_legacy.py to safely merge old databases into the new path. It is idempotent, supports --dry-run, and automatically promotes meaningful memories into the new episodic_memory and working_memory tiers.
    
    2. Tool Call Spam Fixed
    Mnemosyne was auto-logging every terminal, execute_code, write_file, and patch call into memory by default. This flooded working memory with thousands of low-signal entries and made context injection worse for everyone. Auto-logging is now disabled by default (opt-in via MNEMOSYNE_LOG_TOOLS=1). The migration script also includes a --purge-tools flag to clean up existing noise.
    
    Plus:
    - Fixed plugin path resolution in hermes_plugin/__init__.py
    - Registered missing tools in plugin.yaml (triples, sleep, scratchpad)
    - Updated README with upgrade instructions and the new opt-in behavior
    
    Everything is live on main.

    🧠 Mnemosyne just became a monster — here's what we shipped
    
    We went all-in on Mnemosyne this week. The goal was simple: turn it from a "works for me" local memory system into something that can actually compete with Honcho, Mem0, and the rest at scale.
    
    Mission accomplished.
    
    What just landed:
    
    1. FTS5 for working memory — recall() no longer drags 10,000 rows into Python to score keywords in a loop. We built a native fts_working virtual table with SQLite triggers, so working-memory search runs inside SQLite just like episodic memory. Result: from ~5 seconds down to ~6 ms with 10,000 hot memories. That's a 100x speedup.
    
    2. Query embedding cache — Added an LRU cache (512 slots) for query embeddings. In agent loops you often ask the same thing twice. The second time, the embedding is instant — no ONNX overhead.
    
    3. Configurable vector compression — Set MNEMOSYNE_VEC_TYPE to:
       - bit → 32x smaller vectors (~48 bytes each). Millions of episodic memories on a 4GB RAM box.
       - int8 → 4x smaller with barely any accuracy loss.
       - float32 → default, max precision.
       
       sqlite-vec handles the quantization natively via vec_quantize_binary() and vec_quantize_int8().
    
    4. Batch ingestion — New remember_batch() method. We tested 5,000 working-memory inserts in ~0.3 seconds.
    
    5. Hermes plugin hardened — No more sys.path hacks. The plugin now imports mnemosyne cleanly whether it is pip-installed, cloned, or loaded as a skill. We also synced the mnemosyne_native Hermes skill so it rides the same BEAM backend.
    
    Real benchmarks on standard CPU:
    
    | Working memories | Query latency (old) | Query latency (now) |
    |------------------|---------------------|---------------------|
    | 1,000            | ~200-500 ms         | 2.4 ms          |
    | 5,000            | ~1-3 s              | 3.2 ms          |
    | 10,000           | ~5-10 s              | 6.4 ms          |

[2026-04-19T03:22:38.024000+00:00] nulledvector (id=1495263327914426501)
    Mnemosyne v2 is shipped. We just pushed three commits that make local agent memory significantly smarter — without touching cloud infrastructure once.
    
    What's new:
    
    - Recall tracking — memories now know how often they've been accessed and when. Frequently recalled items surface higher; stale ones decay naturally.
    - Recency-weighted scoring — recent context floats up unless high-importance memories override it. No more "why is this 3-month-old memory ranking above what I said yesterday?"
    - Exact-match deduplication — storing the same credential three times is now impossible. remember() updates the existing row instead.
    - Local LLM consolidation — the sleep cycle now uses a 640MB TinyLlama model running entirely on CPU for semantic summarization. Zero API calls. Falls back to AAAK encoding if the model isn't present.
    - Temporal validity — memories can expire. Set valid_until="2026-12-31" and recall() filters it out automatically.
    - Cross-session global memory — remember(..., scope="global") makes preferences travel across every Hermes session. get_context() prioritizes globals so your agent always knows who it's talking to.
    - Invalidation — mnemosyne_invalidate marks outdated facts as superseded. "User moved from Miami to Austin" correctly replaces the old location.
    
    All additive, all backward-compatible, all offline. The DB auto-migrates on startup. If you're running Mnemosyne already, just pull and restart — nothing breaks.
    
    Full details in the README.

[2026-04-22T12:09:26.890000+00:00] nulledvector (id=1496483068704325752)
    here's the long shot: 
    
    Ubuntu 24.04 enforces PEP 668 — it marks the system Python as "externally managed" to prevent pip from conflicting with apt. pip install -e . is blocked by design.
    
    Solutions (best to worst)
    
    Option 1: Use a virtual environment (recommended)
    
    ```
    # Create a venv in the mnemosyne directory
    cd /path/to/mnemosyne
    python3 -m venv .venv
    
    # Activate it
    source .venv/bin/activate
    
    # Now install
    pip install -e .
    ```
    
    Important for Hermes plugins: Hermes needs to run from the same Python environment where Mnemosyne is installed. If you launch Hermes from a different context (e.g., a system-wide installation), it won't see the venv packages. Two ways to handle this:
    
    A) Launch Hermes from the activated venv:
    ```
    source /path/to/mnemosyne/.venv/bin/activate
    hermes  # or however you start it
    ```
    
    B) Install Hermes itself in the same venv, then Mnemosyne:
    ```
    python3 -m venv ~/.hermes/venv
    source ~/.hermes/venv/bin/activate
    pip install hermes-cli  # or however Hermes is installed
    cd /path/to/mnemosyne
    pip install -e .
    ```
    
    Option 2: Use uv (fastest, modern)
    uv respects PEP 668 but handles venvs automatically:
    
    # Install uv
    `curl -LsSf https://astral.sh/uv/install.sh | sh`
    
    # In the mnemosyne directory
    ```
    cd /path/to/mnemosyne
    uv venv
    source .venv/bin/activate
    uv pip install -e .
    ```

    Hey! Sorry for the late reply... Yes, the PyPI install (pip install mnemosyne-memory) is the recommended path now — it handles the Ubuntu/Debian externally-managed-environment issue cleanly via a venv, and you'll get updates automatically.
    
    Here's how to migrate from the old install:
    
    1. Find your old install location(s):
    ```
    # If you cloned the repo
    ls ~/.hermes/projects/mnemosyne
    
    # Or wherever you git cloned it
    ```
    
    2. Remove the old install:
    ```
    # If installed in editable mode (-e)
    pip uninstall mnemosyne-memory
    
    # If symlinked via the deploy script
    rm ~/.hermes/plugins/mnemosyne
    
    # If you have the repo cloned and want to keep it for dev
    # Just leave it — the PyPI install will take precedence in a venv
    ```
    
    3. Install fresh from PyPI (recommended):
    ```
    # Create a venv (avoids Ubuntu 24.04/Debian 12 PEP 668 issues)
    python3 -m venv ~/.venv/mnemosyne
    source ~/.venv/mnemosyne/bin/activate
    
    # Install from PyPI
    pip install mnemosyne-memory[all]
    
    # Verify
    python -c "import mnemosyne; print(mnemosyne.__version__)"
    ```
    
    4. Re-register with Hermes:
    ```
    hermes memory setup
    # → Select "mnemosyne"
    ```
    
    Your data is safe — the database lives at ~/.hermes/mnemosyne/data/mnemosyne.db and is independent of the install method. It'll be picked up automatically.
    
    If you hit any snags let me know! ✨ 🎉

    ## 🧠 Mnemosyne v1.11.0 is live!
    
    Two community-reported bugs squashed. Shoutout to <@199986307324182529>  🫡
    
    ---
    
    🐛 BUG-1: Context overflow on consolidation
    
    Ever run mnemosyne_sleep() and get a wall of truncation warnings? Yeah, that was us. SLEEP_BATCH_SIZE was set to 5000 but the TinyLlama fallback only has a 2048-token context window. We were literally dumping a firehose of memories into a teacup.
    
    Fixed: Token-aware chunking. Now sleep() splits batches to fit the model's context window. Large batches get multi-pass consolidation (chunk summaries → second-pass summary). No more silent truncation, no more garbage summaries.
    
    ---
    
    🐛 BUG-2: No remote/API model support
    
    If you run a bigger model on another machine — llama.cpp server, vLLM, Ollama, whatever — Mnemosyne ignored it completely. The code was hardcoded to local ctransformers/GGUF only. Your 27B Gemma sitting on a box with 262K context? Useless for consolidation.
    
    Fixed: OpenAI-compatible remote client. Three new env vars:
    
    - MNEMOSYNE_LLM_BASE_URL — your server endpoint
    - MNEMOSYNE_LLM_API_KEY — auth token (if needed)
    - MNEMOSYNE_LLM_MODEL — model name
    
    Remote takes priority. Falls back to local ctransformers. Falls back to aaak encoding. Zero new dependencies — httpx if you have it, urllib if you don't.
    
    ---
    
    Grab it: pip install -U mnemosyne-memory
    
    Or pull latest from main.
    
    Questions? Drop them below 👇

    The no such module: vec0 Error
    
    This means sqlite-vec is not loaded into SQLite. The vec0 virtual table module is what sqlite-vec registers. Without it, vector search falls back silently to FTS5 keyword-only search.
    
    Most Likely Cause: Wrong Python Environment
    
    Hermes runs in its own venv at ~/.hermes/.venv/. If sqlite-vec was installed in system Python or a different venv, Mnemosyne won't see it.

    ## v2.3: Mnemosyne can now remember (almost) forever 🧠
    
    I was using [PlanForge](https://discord.com/channels/1053877538025386074/1498331270566903848) to map out the v2.3 feature roadmap when I caught it. The generated document claimed I was building a dashboard with degradation stats. I don't have a dashboard. My own AI tooling had hallucinated a feature into my plan, and my memory system had no way to flag it as fake.
    
    That was the wake-up call.
    
    Every memory in Mnemosyne was stored with the same trust level. User statements, tool injections, agent inferences, imported data, all in one bucket. No way to tell what's real.
    
    So v2.3 ships two things: degradation that lets memories live forever, and confidence that tells you which ones to trust.
    
    How degradation works:
    
    Memories age through three tiers. Hot (under 30 days) keeps full detail. Warm (30-180 days) gets LLM-compressed into summaries. Cold (180+ days) gets reduced to key entities and facts by an entity-density scorer. Your user preferences, the stuff with proper nouns, acronyms, and security terms, keys, survives. The filler doesn't.
    
    How confidence works:
    
    Every memory now carries a veracity tag:
    
    🔹 stated (1.0x) — you said it
    🔹 inferred (0.7x) — the agent guessed it
    🔹 tool (0.5x) — automation injected it
    🔹 imported (0.6x) — came from another platform
    🔹 unknown (0.8x) — unlabeled
    
    The recall engine multiplies by these weights. Tool hallucinations and overconfident inferences get deprioritized automatically. No curation work on your end.
    
    What shipped:
    
    🔹 Three-tier degradation with configurable thresholds
    🔹 Smart compression using entity-density scoring
    🔹 Memory confidence with five veracity levels
    🔹 51 tests, zero regressions
    🔹 All thresholds via env vars, no config files needed
    🔹 pip install mnemosyne — v2.3 live on PyPI

    ```
        The flag exists. Here's the full picture:
        
        Two problems combining to produce the "no_op" result:
        
        1. sleep() is session-scoped — it only consolidates working memories where session_id = ? matches the current session. Your 17 old memories are from past sessions (hermes_20260504_*, etc.), not the current one.

        hermes mnemosyne sleep --all-sessions
            
        This calls sleep_all_sessions() instead, which queries across all sessions (specifically targeting inactive sessions — ones that aren't the current session — for consolidation). Your 17 stale memories from May 4th should get consolidated.
    ```
