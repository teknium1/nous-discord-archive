[2026-05-06T00:32:31.062000+00:00] hackafterdark (id=1501381110527955074)
    You can ask your agent to write scripts because the benchmarks, don't have a universal kinda "API" to hook into. It varies. Basically what that means is the benchmarks need to be able to run the tool calls and most good benchmark tools are doing that not in isolation but through an LLM performing RAG (using the tools). So you're running inference as well (which you could do with a cloud hosted model or a local model).
    
    BEAM is one benchmark that does this and was created by the Hindsight authors because their argument was people were just shoving everything into the context window and as the LLMs got larger context windows and better and better, it was having an impact on the benchmark scores (like LongMemEval). Which is fair (and people have argued this about MemPalace btw). Though you would struggle to run the full BEAM test due to the amount of data involved, you could run it with a smaller data set. Especially if you're using a model that doesn't have a large context window - which addresses the concerns of the authors anyway. For local models and local RAG/memory systems...I don't think BEAM is very suitable. You can run a shorter version of it though.
    
    So the question with any benchmark is, "what are you benchmarking?" In my opinion, it's important to test the entire "agent" system and not just a single component.
    
    For me, RAGAS is the most important benchmark for local agents because it tells you about the accuracy of the data retrieved and also the precision or efficiency. Most people aren't running 1M context windows for local models of course. So I think tokens matter more locally even if they are free, because limited context window and latency. https://docs.ragas.io/en/latest/howtos/cli/rag_eval/

[2026-05-06T10:42:01.832000+00:00] lauratom (id=1501534499454849074)
    HUGE UPDATE: Brainstack v1.0.64 is out.
    
      This is not a small patch. A large chunk of the work since the old public state is now shipped together.
    
      Brainstack has moved from “memory features wired into Hermes” toward a much cleaner Hermes-native memory kernel:
    
      - one canonical behavior-card path, so agent behavior rules do not get duplicated or half-injected
      - stronger separation between truth, support evidence, profile memory, and runtime state
      - tighter Tier2 and background-memory diagnostics, so failures are visible instead of quietly looking active
      - cleaner retrieval and assignment handling with less brittle keyword-style routing
      - fresh Hermes wizard install and Docker build parity verified
      - better bloat, packet, graph, proactive, and source-sync proof surfaces
    
      The main theme of this release is simple:
    
      less memory noise, fewer parallel authority paths, cleaner agent-facing context. (Your agent now has more controll over Brainstack.)
    
      We also took inspiration from mature knowledge-platform designs like WeKnora (a wiki base knowledge system), but kept the Brainstack direction focused: source-backed evidence, admitted truth, compact projections, and Hermes-native agent usability.
    
      Brainstack is starting to feel much less like “extra memory bolted on” and much more like an actual memory kernel for agents.
