    Problem:
    After ~30 turns, context compression silently removes older messages.
    A constraint decided at turn 5 is gone by turn 50.
    The agent contradicts itself and re-asks questions.
    
    **hermes-memory fixes this.**
    Every constraint, decision, and value is stored in SQLite using a terse typed notation (MEMORY_SPEC).
    Context injection stays under 180 tokens regardless of session length.
    Cold facts are retrieved on demand via FTS5.
    
    **What it does:**
    8 MCP tools: write, search, reflect, status, tick, export, purge, optimize
    Hot/cold architecture with automatic pressure management (merge → archive → push to cold)
    Scope lifecycle with scopes opening, closing, and reactivating automatically based on session activity
    Plugin integration with on_session_end hook and /memory slash commands
    memory_optimize migrates MEMORY.md overflow to the DB automatically
    
    52 tests. Zero dependencies. Offline-capable.
    
    PyPI: https://pypi.org/project/hermes-memory/
    Full test : https://github.com/NousResearch/hermes-agent/pull/2692
    PR : https://github.com/NousResearch/hermes-agent/pull/3093

[2026-03-26T09:41:24.159000+00:00] mibayy (id=1486661339459092561)
    SQLite because the whole point is zero infrastructure. No server, no daemon, no credentials — just a file that works anywhere. Postgres requires something running permanently, even locally, which goes against the goal of portability and offline use.
    For vectors, SQLite doesn’t have native support like pgvector. There are extensions like sqlite-vss or sqlite-vec, but they require shipping C extensions, which breaks the “works everywhere without setup” promise. So if vectors matter, Postgres is honestly the better choice.
    
    Indexing local files for Hermes context is a good idea, but it’s a different system. hermes-memory handles structured semantic memory (facts explicitly written by the agent). What you're describing is more of a retrieval layer over a document corpus.
    The typical approach would be: chunk files, embed them (nomic-embed or similar), store them in a vector DB. Postgres + pgvector if you already use it, or something lighter like Chroma or Qdrant. Then retrieve the closest passages at query time and inject them into the prompt. That’s basically what mcp-codebase-index does for code.
    
    My take: don’t try to bolt vectors onto hermes-memory. Keep it focused on structured memory, and build a separate MCP for retrieval. They’ll complement each other naturally.
