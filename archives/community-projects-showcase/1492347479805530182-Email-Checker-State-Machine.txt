    I have no idea if there is a better way to do this. If there is, please let me know. 
    
    ========
    
    Two-tier email processing system for my agent (I gave it Spacemail IMAP via himalaya).
    
    Tier 1 (Dumb): Pure Python script, no LLM. Detects new email, manages state.
    Tier 2 (Smart): LLM session via hermes chat -q. Only invoked when new email detected.
    
    Key goal: Zero LLM calls when inbox is idle.
    
    ┌─────────────────────────────────────────────────┐
    │  CRON: check-email.py (every 1 min)             │
    │  Pure Python, no LLM, ~50ms                     │
    │                                                  │
    │  1. himalaya envelope list                       │
    │  2. Diff against seen_ids.json                   │
    │  3. If .lock exists → log deferred, exit         │
    │  4. If .trigger exists → append new IDs, exit    │
    │  5. New email → write .trigger, log              │
    │  6. Spawn hermes chat -q (background, detached)  │
    └─────────────────────────────────────────────────┘
                          │
                          ▼ (only when new email)
    ┌─────────────────────────────────────────────────┐
    │  LLM SESSION: hermes chat -q "Process email..." │
    │  One-shot, exits when done                      │
    │                                                  │
    │  1. Read .trigger for email IDs                  │
    │  2. Write .lock (email ID + timestamp)           │
    │  3. himalaya message read each email             │
    │  4. Check SPF/DKIM/DMARC headers                 │
    │  5. Reply (trusted+auth) or forward (rest)       │
    │  6. Delete .trigger + .lock                      │
    │  7. Log to events.log                            │
    └─────────────────────────────────────────────────┘

    ## State Files
    
    All in `~/.hermes/email/`:
    
    | File | Purpose | Lifecycle |
    |------|---------|-----------|
    | `seen_ids.json` | Array of processed email IDs | Persistent, updated after processing |
    | `.trigger` | JSON: `{timestamp, ids, summary}` | Created by dumb script, deleted by LLM |
    | `.lock` | JSON: `{email_id, timestamp, pid}` | Created by LLM, deleted by LLM on completion |
    | `events.log` | Timestamped action log | Append-only, rotated manually |
    | `rules.md` | Reply/forward rules | Static, human-editable |
    
    ## Lock Protocol
    
    1. **Dumb script checks .lock first.** If exists → log "deferred, LLM busy", exit.
    2. **LLM writes .lock immediately** on startup (before any email reads).
    3. **LLM deletes .lock + .trigger** on completion (success or failure).
    4. **Stale lock detection:** If .lock is older than 10 minutes, dumb script deletes it and proceeds.
    
    ## Trigger Batching
    
    If multiple emails arrive between checks:
    - First check: email A detected → .trigger created with [A]
    - Second check: email B detected → .trigger exists → append B → .trigger has [A, B]
    - LLM processes both A and B in one session
