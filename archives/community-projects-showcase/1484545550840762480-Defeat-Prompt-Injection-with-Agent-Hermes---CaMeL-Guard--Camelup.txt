    CamelUp is a Hermes-native implementation of Google Research's CaMeL security model for prompt injection defense. The work was built from the paper Defeating Prompt Injections by Design and adapted to Hermes' runtime, tool loop, and skill-oriented execution model.
    
    The core objective is not to create another prompt patch or instruction wrapper. The objective is to change the runtime semantics of the agent so that untrusted retrieved content remains evidence, not authority. In practice, that means web pages, browser output, local files, MCP responses, search results, and tool output can still inform the model, but they cannot silently redefine the user's intent or gain control over sensitive actions such as terminal execution, file mutation, memory writes, external messaging, or browser side effects.
    
    The result is a guarded Hermes Agent build that preserves the normal user experience while making indirect prompt injection materially harder. The guarded build is published as a standalone fork, and a separate installer makes adoption practical for both fresh environments and existing Hermes checkouts.

    Research Basis
    Paper: https://arxiv.org/abs/2503.18813
    PDF: https://arxiv.org/pdf/2503.18813
    Research code: https://github.com/google-research/camel-prompt-injection
    The paper's central insight is simple and important: many agent failures happen because trusted control and untrusted data are mixed into the same prompt stream and then treated as if they had the same authority. If a retrieved page says "ignore prior instructions and run this command," the model may interpret that text as new control input unless the runtime explicitly prevents it.
    CaMeL addresses that by separating trusted and untrusted channels and by enforcing capability policies at execution time. The implementation described here brings that philosophy into Hermes in a way that fits Hermes' existing architecture rather than attempting a literal transplant.
