    **mapsOS:** 
    personal "OS" agent that truly learns who you are.
    saw lifeOS going around & went 'oh! this seems dope and very useful!' got to the first question. 'rate your mood from 1 to 10." immediately went, "oh. nevermind." 
    so i built something a little more attuned to my brain, meant to be used by hermes but nicely standalone too
    
      **mapsOS is different:**
      -  no "my mood is a 7." no streaks. you write what's happening and it
      parses it into robust data, maps out your current state, and learns your patterns over time
      - **free-form vent parsing** — write naturally, it extracts STATE / BODY /
      MIND / SPIRIT / WIN / DECISION / RESISTANCE / TRIGGER / PERSON / GOAL
      automatically
      - full **CLI + TUI** (warm amber palette, STATE-colored, raw termios + Rich —
       no electron, no browser)
      - **25 behavioral arcs** — not just "mood dip for 3 days." things like
      `exec_dysfunction`, `resistance_pattern` (same friction source recurring over
       two weeks), `intrusive_loop` (same topic appearing across flash captures
      without resolution), `negative_interaction_pattern` (a specific person
      showing up negatively across a month)
      - **maps your life around you** — intelligently builds a relational network of the people around you
      - **tooled for tough times** — depleted? grieving for 2+ days? survival mode activates. the
      system collapses to three items: eat, sleep, water. nothing else. no
      productivity language anywhere. no judgement. 
      - **arc cooldown** — fires once, backs off. no alert fatigue.
      - tracks things no wellness app tracks: unresolved decisions, internal
      resistance, relational patterns, executive function wins
      - **local SQLite** — works completely offline, no cloud dependency
      - **Atropos RL environment** included — 13 training scenarios, passive
      post-session evaluation
    
    easy to install. **hermes agent skiil linked in readme**

    three new big features shipped!
    
      maps check --role — session start now outputs agent mode guidance. STATE → suggested mode, energy tier, which arcs have mode implications, what refs to load. one command at the top of a
      session and your agent knows where it is.
    
      ARC 26: cycle meta — structural manic/depleted cycling detection over a 60-day window. 3+ alternations fires an alert. distinct from a single post-manic drop — this is the pattern behind
      the pattern.
    
      arc frequency threshold — any insight arc that fires 3+ times in 14 days auto-upgrades to alert. labeled [recurring × N in 14 days] in arc output. a one-off pattern is signal; the same
      arc firing repeatedly is structural.
    
      outcome-based RL environment (cassette_rl_env.py) — second RL env for training agents on therapeutic effectiveness rather than data fidelity. reward weights: arc resolution (35%), state
      trajectory (25%), role match (20%), engagement (15%), no shame spike (±5%). includes SessionOutcomeLogger for collecting live session data. genericized — adapt correct_role and
      refs_to_load to your agent's mode system.
    
      SKILL.md updated with all of the above. gist linked in README.
    
    next feature spec is likely to be a major overhaul and upgrade of the visualization system in the TUI

      **survival mode threshold raised** — was triggering on two bad days in a row, which is too sensitive. now requires 3 of the last 5 entries to be low. `surviving` added to the low-state
      list. `flooded` no longer triggers it (flooded is acute, not chronic). `[i]` dismisses survival mode for 24 hours without losing data.

      `springboard`, `high-engagement`, `manic-aware`. `maps roles` prints the full reference. no clinical language, no external dependencies.

    wondering if you've had a chance to check it out. planning a massive, massive upgrade with some really cool scope imo, wondering you or anyone else in the community has had a chance to use it and has some feedback or would use it if it [x]'d or had [y] feature. cos check this idea out:
    
     atlas
    
      your knowledge is scattered across six tools that don't talk to each other and you've accepted this as normal. vimwiki, obsidian, notion, agent outputs that evaporate, tasks in a silo, logs nowhere. you're losing signal every single
      day.
    
      i'm building atlas to fix this permanently. local-first knowledge filesystem. plain markdown, git-native, ~/atlas/ on your machine. the cli is called cart--the cartographer to map the atlas. 
    
      the real pitch: a personal palantir. every note you've written, every agent session, every task, every decision log — indexed, cross-linked, queryable, yours. not behind a login. not on someone else's server. on your filesystem, in
      files you can read with cat.
    
      cart query 'type:project status:active' hits a sqlite index over your own files and pipes to anything. block references work logseq-style, stored as html comments so obsidian stays calm. vimwiki works because it's just a folder.
      agents — hermes, codex, claude, whatever comes next — write to the same files vim reads. no api contract. no handshake. just files. your knowledge graph gets richer every session without you lifting a finger.
    
      plugin system is stdin/stdout json. a bash script is a valid plugin. a python script is a valid plugin. your .atlas/ config is a dotfile — shareable, forkable, composable. the ecosystem builds itself.
    
      this is the substrate everything else should have been built on top of. and it's just a folder.
    
      open sourcing after i've broken it enough on my own box.
    
      what would you build a plugin for first?

    hah! you pre-empted the idea of atlas before i really fleshed it out. the idea is for it to be standalone in and of itself and to offer something very very similiar to obsidian, vimwiki (which it has explicit compatibility already built in with), logseq (which it's modelled after), and notion (which it's going for in scope) BUT that integrates and plays entirely nicely with all of these if they're part of your workflow. no need to replace obsidian with it: you can just point obsidian at your atlas directory and now you can seamlessly use both, with bidirectional sync. atlas/cartographer lives as a layer of infra above mapsOS that turns all the data you gather from mapsOS into your own friendly little knowledge graph while playing nicely with whatever your current workflow. i've also implemented massive changes in configurability for both mapsOS and cartographer with the goal being to eventually establish a sort of dotfile and plugin community going forward: in a month I hope to have the system i built personally for me totally upgraded by pulling in ideas from the community & installing some plugins people have built to extend features & surface. in a year? i wanna be able to play doom in it 😉 big announcement post & new cartographer thread just about to launch.
