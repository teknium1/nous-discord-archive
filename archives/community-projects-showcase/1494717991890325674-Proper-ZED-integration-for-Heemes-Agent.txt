[2026-05-10T09:46:23.860000+00:00] nour_h (id=1502970050448195726)
    Prerequisites:
    
    Tailscale running on both your Mac and VPS (sounds like you already have this)
    SSH key-based auth to the VPS (password auth won't work since ACP uses stdin/stdout). If you're using Tailscale SSH, that works too.
    Hermes installed on the VPS (hermes acp must work when you SSH in)
    Step 1 — Verify SSH + Hermes works
    
    From your Mac terminal, test that this connects and prints ACP protocol output:
    
    ```ssh root@<your-vps-tailscale-hostname> "bash -lc 'hermes acp'"```
    
    Replace root with your user if different. Use either the Tailscale hostname (e.g. my-vps) or Tailscale IP (e.g. 100.x.y.z). The bash -lc wrapper ensures your PATH is loaded in the non-interactive SSH session, so hermes is found.
    
    If it prompts for a password, you need to set up SSH key auth first:
    
    ```ssh-keygen -t ed25519  # if you don't have a key yet````
    
    ssh-copy-id root@<your-vps-tailscale-hostname>
    
    Step 2 — Add to Zed settings
    
    Open Zed settings (Cmd+,) and add this to your agent_servers block:
    
    ```
    "agent_servers": {
      "Hermes": {
        "type": "custom",
        "command": "ssh",
        "args": [
          "-o", "StrictHostKeyChecking=accept-new",
          "root@<your-vps-tailscale-hostname>",
          "bash", "-lc", "PYTHONUNBUFFERED=1 hermes acp"
        ],
        "env": {}
      }
    }```
    
    Replace root@<your-vps-tailscale-hostname> with your actual Tailscale user/host.
    
    Step 3 — Use it
    
    Open the Agent panel in Zed, and "Hermes" should show up as an available agent. Select it, and it'll SSH into your VPS transparently each time.
    
    What's happening under the hood: Zed spawns ssh, which connects to your VPS over Tailscale's WireGuard tunnel, runs hermes acp on the remote machine, and pipes the ACP protocol over SSH's stdin/stdout back to Zed. No ports to open, no extra tunnels needed — Tailscale handles the networking, SSH handles the transport.

[2026-05-10T11:24:07.239000+00:00] nour_h (id=1502994643242188821)
    The problem is how SSH handles the command arguments. In your current config:
    ```
    "args": [
      "-o", "StrictHostKeyChecking=accept-new",
      "my_non_root_username@my_tailscale_vps_hostname",
      "bash", "-lc", "PYTHONUNBUFFERED=1 hermes acp"
    ]```
    
    SSH concatenates bash, -lc, and PYTHONUNBUFFERED=1 hermes acp with spaces and sends bash -lc PYTHONUNBUFFERED=1 hermes acp to the remote shell. The remote bash then parses -c as "take the next argument as the command" — which is just PYTHONUNBUFFERED=1 (a no-op env var assignment with no actual command). hermes becomes $0, acp becomes $1, and bash exits immediately with code 0. That's the silent, clean exit Zed is reporting.
    
    The fix — combine the remote command into a single argument so the quoting survives SSH:
    
    ```
    "Hermes": {
      "type": "custom",
      "command": "ssh",
      "args": [
        "-o", "StrictHostKeyChecking=accept-new",
        "-o", "BatchMode=yes",
        "-T",
        "my_non_root_username@my_tailscale_vps_hostname",
        "bash -lc 'PYTHONUNBUFFERED=1 hermes acp'"
      ],
      "env": {}
    }```
