    Hey everyone — the alpha build is live. hipfire is a from-scratch Rust + HIP inference engine for AMD consumer GPUs. No Python in the hot path. On a 5700 XT it runs Qwen3.5-9B at 45 tok/s (vs 4.93 tok/s on llama.cpp+ROCm with the same card).
    
      What it does
      - Qwen3.5 models (0.8B → 27B) with DeltaNet hybrid attention
      - HFQ4 and HFQ6 quantization with hand-tuned RDNA kernels
      - TurboQuant KV cache (FWHT + 4-bit, 7.8× compression for longer context)
      - OpenAI-compatible API server
      - Vision-language support (--image)
    
      Pre-compiled kernels ship for:
      - gfx1010 (RX 5700 XT)
      - gfx1030 (RX 6800 XT)
      - gfx1100 (RX 7900 XTX)
      - gfx1200 (RX 9070)
    
      If your GPU isn't on that list, there's an agent skill (.skills/hipfire-tester/) that works with Hermes or any agent — it can detect your GPU, compile kernels for your specific arch, configure the engine, and walk you through fixing any errors that come up. Just point your agent at the repo and it has the full context.
    
      # Pull a model and go (works similar to ollama) 
      hipfire pull qwen3.5:9b
      hipfire pull qwen3.5:9b         # 4.5GB, 45 tok/s, best quality (8GB)
      hipfire pull qwen3.5:27b        # 14.3GB, needs 16GB+ VRAM
      Add -hfq6 for 6-bit (higher quality, ~15% slower): hipfire pull qwen3.5:9b-hfq6
    
    
      Links
      - GitHub: https://github.com/Kaden-Schutt/hipfire (branch: alpha-builds)
      - Models: https://huggingface.co/schuttdev/hipfire-models
      - 27B model: https://huggingface.co/schuttdev/Qwen3.5-27B-hipfire
      - Contributing/benchmarks: CONTRIBUTING.md in the repo
    
      Benchmarks from cards other than the 5700 XT are the main thing I need right now. If you run it, drop your GPU + model + tok/s here or open an issue. The bench script auto-detects your VRAM and skips models that won't fit: bash scripts/megabench-q35.sh

    Your GPU (RX 9070XT - gfx1200) is supported! The guide lists it as an alpha target with 16GB VRAM.
    
    Key optimizations that could help you hit >50 TPS:
    
    1. HFQ4/HFQ6 quantization formats - These are hipfire-specific formats:
       - HFQ4 (0.53 B/w): Fastest, good quality
       - HFQ6 (0.78 B/w): Better quality, slightly slower
       
    2. TurboQuant KV compression:
       - --turbo4: 7.8x KV compression, ~97% speed retention
       - --turbo2: 15.5x KV compression, ~94% speed retention (good for ≤4B models)
    
    3. No Python overhead - The entire inference path is Rust-native HIP, which should eliminate the Python hot path bottleneck you're hitting with llama.cpp
    
    Your Qwen3.5-27B would need:
    - HFQ4: ~14.3GB VRAM (fits on your 16GB card)
    - HFQ6: ~21.1GB VRAM (too large, use --turbo4 for KV cache reduction)
    
    Next steps if you want to try this:
    1. Clone and build hipfire
    2. Quantize Qwen3.5-27B to HFQ4 format

    1. Don't strip think tags — extract the post-think content instead
    Change your serve_wrapper strip logic from "remove everything between <think>...</think>" to "return everything after </think>":
    pythonimport re
    
    def strip_think(text: str) -> str:
        # Return only the content after </think>, not the whole block removal
        match = re.search(r'</think>(.*)', text, re.DOTALL)
        if match:
            return match.group(1).strip()
        return text  # no think tags, return as-is
    This is the minimal fix and works immediately.
    2. Force /no_think in the system prompt
    Qwen3.5-based models accept a special token. Try prepending to your system prompt:
    /no_think
