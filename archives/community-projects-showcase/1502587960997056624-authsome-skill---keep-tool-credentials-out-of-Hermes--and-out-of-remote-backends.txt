    Hey
    
    I am maintainer of authsome here, posting because Hermes' built in credential pool is good for LLM provider keys (rotation, 401 refresh, least-used selection), but the keys for the long tail of tool providers (Stripe, GitHub, Notion, Resend, Linear) still live in the Hermes process. 
    
    fine on a trusted laptop. Less fine when Hermes runs on modal, daytona, ssh, or a $5 VPS you talk to from Telegram, because any LLM-chosen tool in Hermes' address space can read them.
    
    authsome-skill wires Hermes through authsome, a local open-source credential broker (https://github.com/manojbajaj95/authsome, MIT).
    
     it works like this:
    first:
    
    Credentials live in a local SQLite vault under ~/.authsome on your machine.
    
    Hermes' tool-layer HTTP calls route through a local proxy on 127.0.0.1, the proxy attaches the real header on the way out, the agent's env shows authsome-proxy-managed.
    
    For remote backends (Modal, SSH, Daytona), the skill tunnels Hermes' outbound traffic back to your local authsome, so cloud VM holds zero credentials.
    
    What it is not supposed to be
    1. Not a replacement for Hermes' LLM provider rotation, that stays.
    2. Not a competitor to hermes-vault or ClawShell. 
    
    Different shapes, both fine, keep using them. authsome is the lighter option with 13 OAuth2 and 31 API-key providers preconfigured.
    
    `hermes skills install manojbajaj95/authsome/skills/authsome`
    
    
    Anyone running Hermes on a remote backend, does the local-vault-with-tunnel pattern fit, or is the latency a problem under parallel subagent load.
    
    hermes-vault and ClawShell users, where does authsome fall short for you.
    
    Providers I should preconfigure that I have not.
