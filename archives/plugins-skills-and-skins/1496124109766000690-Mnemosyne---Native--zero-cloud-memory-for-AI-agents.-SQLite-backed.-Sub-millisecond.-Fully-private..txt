[2026-04-25T17:50:53.325000+00:00] nulledvector (id=1497656158637391968)
    Our retrieval system is built on three layers that work together to find what matters across sessions. At the core is hybrid search that blends vector similarity, full text search, and importance scoring into a single ranked result set. Vector search handles semantic meaning even when wording changes between sessions. Full text search catches exact phrases and keywords. Importance scoring learned from your behavior boosts memories you recall frequently. Below that sits the BEAM architecture with working memory for hot recent context and episodic memory for long term storage. Old working memories automatically consolidate into episodic summaries through sleep cycles, so nothing gets lost to time. The entire system runs on SQLite with sub millisecond latency, no network calls, no rate limits, and no vendor lock in. Cross session recall is opt in via scope equals global, which keeps privacy first by default rather than blindly persisting everything. The honest caveat is that retrieval quality depends on what you mark global and our 384 dimension embeddings are smaller than cloud alternatives. If you want zero config cloud memory, Honcho is less work. If you want fast, private, controllable cross session recall that learns what matters to you, Mnemosyne wins.

[2026-04-26T19:05:24.168000+00:00] nulledvector (id=1498037298577805494)
    You don't need to disable built-in memory for Mnemosyne. The Hindsight docs suggest that because their vector DB replaces the same function, so you'd get duplicate context otherwise.
    
    With Mnemosyne it depends on what you want:
    
    - Keep both on (default): agent sees MEMORY.md + Mnemosyne recall results. More context, no conflict. They're additive, not competing.
    - Disable built-in memory toolset (hermes tools disable memory): agent can't edit MEMORY.md anymore, only uses Mnemosyne. Cleaner if you want Mnemosyne as your single source of truth.
    - Disable injection too (memory_enabled=false, user_profile_enabled=false): agent stops seeing MEMORY.md/USER.md entirely. Only Mnemosyne feeds it memories.
    
    The reason it's confusing is there's no smart switch. It's 4 independent toggles that stack:
    
    1. memory_enabled / user_profile_enabled → injects text into prompt
    2. memory toolset → gives agent the tool to edit those files
    3. memory.provider → routes mnemosyne_recall/mnemosyne_remember to Mnemosyne
    4. skip_memory=True → kills everything
    
    They don't override each other. They just pile on. So pick what fits your setup and ignore the rest.

[2026-04-27T19:15:05.190000+00:00] nulledvector (id=1498402123426037772)
    Mnemosyne is built for bare minimum CPU.
    
    Embedding model: Default is BAAI/bge-small-en-v1.5 via fastembed (ONNX runtime, no PyTorch, no GPU). 384 dimensions. Falls back to FTS5-only if fastembed isn't installed. You can plug in your own via env vars? Let me check:
    
    The embeddings module hardcodes _DEFAULT_MODEL = "BAAI/bge-small-en-v1.5" with no override mechanism. The local_llm module does have MNEMOSYNE_LLM_REPO and MNEMOSYNE_LLM_FILE env vars for swapping the consolidation model, but embeddings has no equivalent.
    
    LLM for consolidation: Default is TinyLlama-1.1B-Chat Q4_K_M (~600MB) via ctransformers (GGUF, CPU-only). Also supports remote OpenAI-compatible endpoints via MNEMOSYNE_LLM_BASE_URL, MNEMOSYNE_LLM_API_KEY, MNEMOSYNE_LLM_MODEL.
    
    Architecture support: ctransformers uses llama.cpp under the hood, so anything GGUF supports: x86_64, ARM64 (Apple Silicon, Raspberry Pi), etc. fastembed/ONNX runs on the same.

    Options:
    
    1. Upgrade the consolidation LLM. Mistral 3 14B is the bottleneck. Switch to something stronger for consolidation:
       - Qwen 2.5 32B (good at structured output)
       - Llama 3.1 70B (better summarization)
       - Even GPT-4o-mini via API for consolidation only (cheap, fast, good summaries)
    
    2. If you must keep Mistral 3 14B, reduce consolidation pressure.
       - Smaller batches = less content per summary = less compression loss
       - More frequent consolidation = memories are fresher when summarized
    
    ```
    MNEMOSYNE_SLEEP_BATCH=50    # Default is 5000, way too big for weak models
    MNEMOSYNE_WM_TTL_HOURS=12   # Consolidate faster, while memories are fresh

[2026-05-07T02:48:04.485000+00:00] nulledvector (id=1501777612442435665)
    MNEMOSYNE_LLM_ENABLED gates a single shared backend (local_llm.py) that both consolidation and fact extraction use. When enabled with no remote API configured, TinyLlama handles both tasks. Consolidation works fine because summarization is forgiving and falls back to aaak encoding if the model fails. Fact extraction does not work fine... TinyLlama cannot reliably produce structured JSON, and the extraction pipeline silently returns empty fact lists when the LLM output is unparseable. So no, TinyLlama is not sufficient as a drop-in replacement if you were depending on Mistral for extraction. The two tasks share infrastructure but have very different quality requirements.

[2026-05-10T08:48:07.979000+00:00] _voscko_ (id=1502955387660271637)
    Yeah, that makes sense. If it was Hermes "state.db", I’d treat this mostly as a Hermes/session-storage boundary issue, not necessarily a Mnemosyne bug.
    
    Hermes may handle normal Telegram/Discord attachments fine by caching them as files and passing local paths around, but "state.db" still stores the actual session messages/tool payloads. So base64 can still sneak in if it arrives as text, an OpenAI-style "data:image/...;base64,..." image_url, a tool result, an API payload, or some plugin/adapter path that bypasses the normal attachment cache.
    
    So I think the fix is probably not “Mnemosyne should store images differently” only. It is more like: anything before SQLite should sanitize large/binary-looking payloads and replace them with artifact refs.
    
    Mnemosyne can still add guardrails on its own "remember"/import paths, but Hermes "state.db" needs protection too because it is upstream and stores the full session history.

[2026-05-10T12:12:36.413000+00:00] nulledvector (id=1503006845202337893)
    Mnemosyne ships an MCP server since v1.13.0. That means any agent in your pipeline - orchestrator, coder, reviewer - just points its MCP config at Mnemosyne.
    
    The v2.1 identity layer means coder-A doesn't pollute coder-B's recall, but cross-agent relevance still surfaces when it should. So the orchestrator stores a design decision once, and every coder who later picks up a related task gets it injected automatically.
    
    LLMWiki is a markdown wiki agents have to read. 
    
    Mnemosyne is institutional memory that accumulates passively - every bug fix, every "don't do X here", every architectural call gets stored and recalled when relevant. Agents don't read docs. They just work, and the right context finds them.
    
    In a Kanban pipeline where 5+ specialized agents are spinning up and down, that's the difference between a curated library nobody reads and a living memory that makes every agent smarter than the last one.
