    brainctl is a local-first agent memory layer. One SQLite file, FTS5 search, optional vector embeddings, knowledge graph, and a full temporal cognition stack (epochs, temporal classes, recency-weighted recall, causal threading, session-start orientation). No API keys, no server, no LLM calls. Pip-installable, v1.1.2 on PyPI.
    
    Why brainctl: the failure mode we kept hitting was agents waking up with recall but no sense of time — a three-month-old decision felt as urgent as a three-hour-old one. brainctl fixes that: every memory has a temporal class (permanent → ephemeral), recall is recency-weighted per scope, events are causally linked, and agents get a compact "where am I in the story" block at session start. It stops agents from re-litigating settled decisions or acting on stale context.
    
    Two clean integration paths:
    • Python API — from agentmemory import Brain; brain = Brain(agent_id="hermes"). ~10 lines to wire up.
    • MCP server — brainctl-mcp exposes ~192 tools over stdio. If Hermes speaks MCP, it's a config entry.
    
    Onboarding is accounted for: 5-min quickstart in docs/AGENT_ONBOARDING.md, brain.doctor() health check, brain.handoff() / brain.resume() for session continuity, W(m) write gate against memory bloat, append-only migrations. Plugin extras already exist for LangChain and CrewAI — a brainctl[hermes] is a one-liner.
    
    What I'd want back: guidance on which integration path fits the plugin model better, how Hermes expects plugins to declare agent identity, and any session-boundary conventions.
    
    Happy to open a PR, write the Hermes extra, or pair with whoever owns memory architecture. MIT, single-file DB, trivial to audit.

    Hey there thanks for the ping.... brainctl is more like a swiss army knife. It has a web UI, MCP server, connects to LangChain and CrewAI, and uses around 12 tables. Its meant to be a universal memory system for pretty much any agent out there.
    
    Mnemosyne is way lighter. Its just 3 SQLite tables, runs on anything including old hardware, and its built specifically for the Hermes ecosystem. No extra installs, no GPU needed.
    
    So yeah, same general idea but totally different approaches. brainctl goes big, Mnemosyne keeps it small footprint. Both work fine, just depends what you need. Hope I helped in something! ✨

    I finally succeeded in installing brainctl into Hermes but there are some issues in achieving this - both Hermes issues & brainctl issues:
    Third-party memory providers should install to ~/.hermes/plugins/<name>. Current Hermes releases have a discovery bug where hermes memory setup/status scan only the bundled plugins/memory/ tree, so affected versions require a temporary symlink from ~/.hermes/hermes-agent/plugins/memory/<name> to ~/.hermes/plugins/<name>. Plugin Python dependencies must be installed into the Hermes virtual environment because Hermes imports them with its own interpreter.
