    We built an RTK integration for Hermes that transparently rewrites terminal commands to their RTK equivalents, saving 60-90% of tokens in your context window.
    
    How it works
    When the agent runs git status, the plugin intercepts the pre_tool_call hook, delegates to rtk rewrite, and the command becomes rtk git status. The compressed output reaches the LLM instead of raw verbose output. 30+ commands supported — git, grep, find, ls, cargo, pytest, npm, docker, kubectl, and more.
    
    Install
    
    Step 1 — Install RTK:
    # macOS
    brew install rtk
    
    # Linux / macOS (alternative)
    curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh
    
    
    Step 2 — Install the plugin:
    pip install rtk-hermes
    
    
    That's it. The plugin auto-registers via the hermes_agent.plugins entry point — no config needed. Just restart Hermes.
    
    Measured savings
    - cargo test — 90-99%
    - git log --stat — 87%
    - ls -la — 78%
    - git status — 66%
    
    Graceful degradation — if RTK isn't installed or crashes, commands pass through unchanged. The plugin never blocks execution.

[2026-05-04T09:25:53.548000+00:00] vgallotti (id=1500790562951790743)
    Thanks for the detailed report. You were not missing RTK — this was an install-path issue in my docs.
    
    `$(dirname "$(which hermes)")/python` only works when the `hermes` executable lives directly inside the same virtualenv as its Python binary. In your case, `hermes` is exposed through `~/.local/bin/hermes`, so that command incorrectly looked for `/home/<user>/.local/bin/python`.
    
    For the standard Hermes source install, use the Hermes virtualenv directly:
    
    bash
    HERMES_PY="$HOME/.hermes/hermes-agent/venv/bin/python"
    "$HERMES_PY" -m pip install --upgrade rtk-hermes
    
    If that returns `No module named pip`, the Hermes virtualenv exists but was created without pip. On Debian/Ubuntu, don't use plain `pip` and don't use `--break-system-packages`; that hits the system Python and triggers PEP 668. Use `uv` to install into the Hermes virtualenv instead:
    
    bash
    curl -LsSf https://astral.sh/uv/install.sh | sh
    ~/.local/bin/uv pip install --python "$HOME/.hermes/hermes-agent/venv/bin/python" --upgrade rtk-hermes
    
    Then enable the plugin in `~/.hermes/config.yaml`:
    
    yaml
    plugins:
      enabled:
        - rtk-rewrite
    
    Restart Hermes or start a new session after that.
    
    I pushed `rtk-hermes` `v1.2.2` with updated README/install docs covering this exact Debian/PEP 668 case:
    https://github.com/ogallotti/rtk-hermes/releases/tag/v1.2.2
    [embed] Release rtk-hermes v1.2.2 · ogallotti/rtk-hermes https://github.com/ogallotti/rtk-hermes/releases/tag/v1.2.2
