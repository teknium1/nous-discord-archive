[2026-04-07T14:24:08.218000+00:00] w0lfgang88 (id=1491081146266681414)
    Built a standalone plugin that replaces the built-in context compressor with DAG-based summarization. Every message preserved, nothing lost to compaction.
    
    ## What it does
    
    When context fills up, Hermes replaces middle turns with a flat lossy summary. LCM replaces that with:
    
    - **Immutable message store** — every message persisted verbatim in SQLite
    - **Hierarchical summary DAG** — D0 (minutes) → D1 (hours) → D2 (days)
    - **3-level escalation** — detailed → bullets → deterministic truncate (guaranteed convergence)
    - **Agent tools** — `lcm_grep`, `lcm_describe`, `lcm_expand` so the agent can drill back into anything that was compacted
    
    ## How is this different from Honcho / Mem0 / RetainDB?
    
    Different layer. Those are memory providers — they add cross-session recall. LCM replaces the context engine itself — how the active window is managed when it fills up. They stack, they don't compete.
    
    ## Required a core patch
    
    Hermes doesn't have a context engine plugin slot, so I built one. Small PR — adds a ContextEngine ABC that makes the compressor swappable via the plugin system. Same pattern as OpenClaw's contextEngine slot + lossless-claw.
    
    ## Install
    
    Requires the plugin slot PR to be merged first, then:
    ```
    git clone https://github.com/stephenschoettler/hermes-lcm ~/.hermes/plugins/hermes-lcm
    ```
    Restart Hermes. That's it.

[2026-04-17T10:25:52.724000+00:00] _voscko_ (id=1494645065275146310)
    Short version: no, LCM does not replace Hermes memory providers, and it does not conflict with session_search.
    
    LCM is a context engine plugin. Its job is to replace the default compaction/compression path so old conversation detail is preserved in its own DAG/store and can be retrieved later with tools like lcm_grep, lcm_describe, lcm_expand, and lcm_expand_query.
    
    That is a different layer from memory providers. So:
    
    - MEMORY.md / USER.md still exist as Hermes built-in prompt memory if you have built-in memory enabled
    - Hindsight is still useful if you want richer long-term memory / recall across sessions
    - LCM and Hindsight can be used together because they solve different problems
    
    A simple way to think about it:
    
    - built-in memory / Hindsight = “what stable facts should Hermes remember?”
    - LCM = “how do we preserve and recover detailed conversation context when the chat gets compacted?”
    - session_search = “search the archived Hermes transcripts”
    
    On session_search specifically: no direct conflict. session_search still works against Hermes’ normal session archive. LCM adds its own retrieval tools over the LCM store. There is some conceptual overlap, but they are separate mechanisms and separate tool names.
    
    So no, you do not need Hindsight just to “make LCM work.” But yes, you may still want Hindsight if you want better long-term memory handling than MEMORY.md / USER.md alone.
    
    Josh’s setup makes sense:
    - Hindsight for persistent memory
    - LCM for context continuity and compaction recall
    - LLM-wiki for long-running structured projects
    
    That stack is actually pretty reasonable if you bounce between platforms and juggle multiple unrelated threads.

[2026-04-17T12:36:50.736000+00:00] _voscko_ (id=1494678024166314127)
    That log line is informational, not an error. Are you on the latest hermes agent update? It just means your Hermes host supports the LCM context engine, but not plugin slash-command registration, so /lcm is unavailable on that host. The plugin itself can still be working fine.
    
    A few good ways to verify it:
    
    1. Check that Hermes sees the plugin and that lcm is the active context engine: hermes plugins
    
    You want to see something like:
    - hermes-lcm listed as installed
    - Context Engine: lcm
    
    2. Make sure your config has:
    ```context:
      engine: lcm```
    
    
    3. Look for the positive log line:
    LCM plugin loaded — lossless context management active
    
    4. Even if /lcm is unavailable, the LCM tools should still work. You can ask Hermes to use:
    - lcm_status
    - lcm_doctor
    - lcm_grep
    
    For example:
    - “Use lcm_status and tell me what it shows.”
    - “Use lcm_doctor and report any problems.”
    
    If those tools respond normally, the plugin is loaded and working.
    
    So in short:
    - no /lcm on that host = older / different Hermes host capability
    - LCM itself can still be active and working correctly
    - hermes plugins + lcm_status / lcm_doctor are the best checks

    # PR: Fix LCM context engine missing `on_session_end` in ACP mode
    
    **Repository:** https://github.com/NousResearch/hermes-agent  
    **Branch:** `fix/acp-lcm-on-session-end`  
    **File changed:** `acp_adapter/server.py`  
    **Labels:** `bug`, `acp`, `lcm`, `context-engine`
    
    ---
    
    ## Summary
    
    When Hermes runs in ACP (Agent Communication Protocol) mode (e.g. via multica, VS Code, Zed, JetBrains), the LCM (Lossless Context Management) context engine's `on_session_end()` hook is never called at the end of a `session/prompt` turn. This causes all conversation messages to remain un-ingested in the LCM store (`~/.hermes/lcm.db`), making them invisible to LCM tools (`lcm_grep`, `lcm_expand`, `lcm_describe`).
    
    The Hermes CLI/TUI correctly calls `context_compressor.on_session_end()` on session boundaries (exit, `/new`, `/reset`). The ACP adapter only calls `session_manager.save_session()` (which writes to `state.db`), omitting the LCM flush entirely.
    
    ## Problem Details
    
    ### Hermes' four-layer memory architecture
    
    1. **session_search** — full-text search over `state.db` ✅ works in ACP
    2. **LCM** — DAG-based context compression & retrieval over `lcm.db` ❌ **broken in ACP**
    3. **skills** — static skill index ✅ works in ACP
    4. **memory** — persistent MEMORY.md/USER.md ✅ works in ACP
    
    ### Root cause
    
    In `acp_adapter/server.py::prompt()`, after `agent.run_conversation()` returns:
    
    ```python
    if result.get("messages"):
        state.history = result["messages"]
        # Persist updated history so sessions survive process restarts.
        self.session_manager.save_session(session_id)  # ← only state.db
    ```
    
    `save_session()` calls `SessionDB._persist()` which writes messages to `state.db`. However, it does **not** call `agent.context_compressor.on_session_end()`, which is required by LCM to ingest pending messages into `lcm.db`.

[2026-05-02T11:43:14.906000+00:00] _voscko_ (id=1500100353981546526)
    On retention: it is not only the in-memory user session.
    
    hermes-lcm stores its own SQLite DB, profile-scoped by default at `~/.hermes/lcm.db`, so the raw message store and DAG summaries survive normal restarts. The main retrieval path is intentionally scoped around the active/current session, because LCM is meant to protect the conversation the agent is actively compacting, not become a global memory system.
    
    On `/new`, retained summary nodes can be carried forward depending on `LCM_NEW_SESSION_RETAIN_DEPTH`. The default is `2`, which means higher-level summaries are kept, while lower-level session detail can be pruned. You can set it to `-1` to keep all DAG nodes, or `0` to keep none across `/new`.
    
    So the short version is: the storage persists beyond the process/session, but the active recall behavior is session-oriented by design.
    [reply to kohai_ut msg=1499839628566597833]

[2026-05-02T11:47:24.927000+00:00] _voscko_ (id=1500101402645762238)
    For aux settings, I would start boring: leave `LCM_SUMMARY_MODEL` and `LCM_EXPANSION_MODEL` unset and let Hermes use its configured auxiliary model.
    
    If you do override them, I’d optimize for reliability and summarization quality more than raw speed. Bad summaries are worse than slightly slower summaries, because they become the index the agent relies on later.
    
    My rough recommendation:
    
    Use a solid, not tiny model for `LCM_SUMMARY_MODEL`. It does not need to be your best reasoning model, but it should be good at preserving names, decisions, constraints, code references, and tool results.
    
    Use the same or a stronger model for `LCM_EXPANSION_MODEL` if you rely on `lcm_expand_query`, because that path synthesizes an answer from retrieved context.
    
    I would only split cheap summary model plus stronger expansion model after dogfooding it a bit. The default auxiliary path is the safest first setup.

    Same broad problem, different layer.
    
    **Honcho** is memory:
    
    * cross-session recall
    * semantic retrieval
    * observations/conclusions about users/projects
    
    **LCM** is context management:
