    🛑 My Hermes agent used to start executing the moment I hit enter.
    
    I'd describe a vague idea and it would immediately start writing files, calling tools, making changes I hadn't fully thought through yet.
    
    So I built converse mode. One plugin, two commands. The agent won't touch a single tool until you say so.
    
    **The problem it solves:**
    
    Agents are fast. Too fast. You type half an idea and they're already three steps deep. What you actually need is a thinking partner that asks the right questions before it does anything.
    
    **How it works:**
    
    `/converse` flips the agent into discussion-only mode. It asks clarifying questions, surfaces tradeoffs, pushes back on unclear requirements. Tools are blocked at the hook level. It physically cannot execute even if the model tries.
    
    `/go` flips execution back on with the full conversation baked into context. No re-explaining. No re-scoping.
    
    **Every response in converse mode ends with:**
    
    ```
    Plan so far:
    GOAL: ...
    IN SCOPE: ...
    OUT OF SCOPE: ...
    OPEN QUESTIONS: ...
    NEXT STEPS: ...
    ```
    
    You always know exactly where the conversation has landed before you pull the trigger.
    
    **What you get:**
    
    🔒 Tool calls blocked at hook level. Model can't sneak past it.
    🧠 Pre-LLM injection. Scoping instructions baked in before every call.
    📋 Structured plan recap after every turn
    ⚡ Zero external dependencies. Pure Python, no installs.
    🔧 Two hooks: `pre_llm_call` + `pre_tool_call`
    🧵 Thread-safe, per-session. Works across concurrent users.
    
    **Install in two lines:**
    
    ```bash
    hermes plugins install ibrahimokdadov/hermes-plugin-converse
    hermes plugins enable converse
    ```
    
    Then:
    
    ```
    /converse    → start scoping, tools blocked
    /go          → execute with full context
    ```
    
    MIT license. No subscriptions. No network calls. The agent just waits until you're ready.

    Hey, had my HA look this over. This is the type of functionality/mode I find useful! This is what the agent reported. Grain of salt and all that. I’m going to give it a shot regardless!
    
    ---
    
    Solid idea but has some issues:
    
    `_ctx_ref` is global — last session to register wins. if you're running the gateway with multiple sessions active, `/go` in one session will inject into whoever registered last. race condition that'll bite you in multi-user setups.
    
    session IDs also never get cleaned up from the set if the session dies unexpectedly. no hook for that, so it'll grow unbounded if people use converse and then crash or force close.
    
    `/go` also fires regardless of whether converse mode was ever on, which could be confusing.
    
    and `inject_message` is called with `role="user"` but nothing confirms that API actually accepts that kwarg — could be silently failing.
    
    overall it's a good pattern and the README is well written, but the ctx ref thing is a real bug worth fixing before anyone depends on it in a gateway environment.
