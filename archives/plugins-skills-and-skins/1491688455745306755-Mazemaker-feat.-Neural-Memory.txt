    ## Features (From first proof of concept builds - more below)
    
    - **Semantic Memory Storage**: Store memories with automatic embedding generation (`BAAI/bge-m3`, 1024d, CUDA-accelerated)
    - **Knowledge Graph**: Automatic connection of related memories based on cosine similarity
    - **Spreading Activation**: Explore connected ideas through BFS graph traversal with decay
    - **Conflict Detection**: Automatically detect and supersede conflicting memories
    - **Dream Engine**: Autonomous background consolidation (NREM/REM/Insight phases)
    - **LSTM+kNN Pattern Learning**: C++ LSTM predicts next-relevant embedding from access sequences; kNN re-ranks by multi-signal score (embedding + temporal + frequency + graph)
    - **MSSQL Backend**: Optional shared database for multi-agent setups — all dream operations write to the `connections` table directly
    - **CUDA Acceleration**: GPU-accelerated embeddings via sentence-transformers

    --------------
    **Me:** so are you thinking two brains? one for structured knowledge and another for evo and then maybe one manages the other? evo manages the wiki?
    
    **Agent: **That's a pretty sharp mental model actually. Think of it like human memory:
    
    Neural Memory (Evo) = Hippocampus
    - Catches everything, makes associative connections automatically
    - Spreading activation — "chicken or fish" surfaces related memories about email, himalaya, food preferences
    - Fast, fuzzy, decays over time (24hr half-life)
    - Working memory — what's relevant right now
    
    LLM Wiki = Neocortex
    - Consolidated, structured, permanent
    - Cross-linked and curated — every page is readable in 30 seconds
    - The "source of truth" that survives sessions
    - Auditable — you can read it yourself, edit it, trust it
    
    The flow:
    Incoming (email, chat, research)
            ↓
       Neural Memory catches it
       (auto-embed, auto-connect, activate related memories)
            ↓
       Agent decides: is this worth consolidating?
            ↓
       Wiki gets updated (entity/concept pages, cross-references)
            ↓
       Wiki feeds back into neural memory as source material
    
    
    So Evo doesn't just "manage" the wiki — it's the front door. Everything flows through associative recall first, gets filtered by relevance, then the durable stuff gets written up properly into the wiki.
    
    The wiki lint tasks (orphan detection, contradiction flagging, staleness) could also be driven by Evo's conflict detection — it already knows which memories contradict each other.
    
    It's basically a two-tier memory system: fast/associative/decaying + slow/structured/permanent. With the agent as the consolidation mechanism (like sleep does in humans).

[2026-04-11T10:26:50.435000+00:00] alcatr4z (id=1492470980138893413)
    But... Good catch, thought i got it,... Another fix pushed:
    
    The problem was:
    Before: prefetch() consumed _initial_context (= deleted it).
    
    Turn 1: _build_system_prompt → Context is in cache ✓
    Turn 1: prefetch() → Context consumed, deleted
    System Prompt cache is not updated → Turn 2+ still have Context
    But: Context was ONLY in the System Prompt, not in the User Message
    
    After:
    
    prefetch() no longer consumes — _initial_context stays available
    Context is injected twice:
    
    In System Prompt (system_prompt_block() → ## Recent Memory Context)
    In User Message (prefetch() → build_memory_context_block() fenced block)
    
    
    
    Double-injection = Model sees the context in both places. Models tend to ignore the System Prompt but usually read User Message content more carefully.

    Neural Memory — What Broke & What We Fixed
    
    Root cause: Running an outdated version of the plugin + wrong DB path in config.
    
    Issues found:
    
    Wrong DB path — Config pointed to ~/.neural_memory/memory.db (empty, 0 rows). Actual data lived at ~/.hermes/hermes-agent/~/.neural_memory/memory.db (148 memories, 628 connections). The plugin was reading/writing to the correct path internally, but the config-level path was stale — so dream auto-triggers and dashboard generation couldn't find the data.
    
    Dream engine never auto-triggered — Should fire after 5 min idle or every 50 new memories. Only ran once in its entire lifetime (143+ memories). Likely caused by the daemon watching the wrong DB file. Fixed by adding a cron job (0 */6 * * *) as the reliable trigger.

    Deep dream (every 10th cycle does full-history sampling to prevent long-term erosion)
